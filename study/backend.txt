----------------------------------------------------------------
2. core/ - 공통 설정
----------------------------------------------------------------

[config.py]
  - python-dotenv로 .env 파일을 읽어 환경 변수를 상수로 노출한다.
  - SUPABASE_URL, SUPABASE_KEY가 없으면 앱 시작 시 RuntimeError로 즉시 종료.
  - GEMINI_API_KEY는 없어도 경고만 출력 (RAG 기능만 비활성화됨).
  - MAX_FILE_SIZE = 50MB 상수 정의.

[supabase.py]
  - create_client()를 호출해 supabase 클라이언트를 모듈 수준(전역)에서 1개만 생성.
  - 모든 service 파일에서 'from app.core.supabase import supabase'로 가져다 씀.

[gemini.py]  ← 신규
  - Gemini 클라이언트(genai.Client)와 모든 모델/파라미터 상수를 한 곳에서 관리.
  - 상수 목록:
      CHAT_MODEL            = "gemini-2.5-flash-lite"
      REWRITE_MODEL         = "gemini-2.5-flash-lite"
      EMBEDDING_MODEL       = "gemini-embedding-001"
      EMBEDDING_DIMENSIONS  = 1536
      CHUNK_SIZE            = 1000   (RAG 청킹 단위, 문자 수)
      RAG_CHUNK_LIMIT       = 6     (벡터 검색 반환 청크 수)
      HISTORY_LIMIT         = 6     (대화 히스토리 로드 개수)
      REWRITE_HISTORY_WINDOW= 4     (쿼리 재작성에 사용할 최근 대화 수)
  - document.py, chat.py 모두 여기서 import.
  - 모델을 교체할 때는 이 파일만 수정하면 됨.

3. models/ - 데이터 형식 (Pydantic)
----------------------------------------------------------------

[auth.py]
  - RegisterRequest  : student_id, login_id, password, name (회원가입 입력)
  - LoginRequest     : login_id, password (로그인 입력)
  - UpdateUserRequest: student_id, login_id, name, password (모두 Optional)
  - UserResponse     : id (UUID), student_id, login_id, name (응답 - 비밀번호 제외)
    ※ id는 UUID 문자열. 프론트 localStorage에 저장되어 API 호출 시 사용됨.

[category.py]
  - CategoryCreate   : user_id, name (카테고리 생성 입력)
  - CategoryResponse : id, user_id, name, created_at (응답)

[document.py]
  - UploadResponse   : document_id, file_path, original_file_name (업로드 직후 응답)
  - DocumentResponse : id, user_id, category_id, original_file_name,
                       parsing_status, created_at (목록 조회 응답)

[chat.py]
  - ChatSessionCreate   : user_id, title
  - ChatSessionResponse : id, user_id, title, created_at
  - ChatAskRequest      : session_id, content, document_ids(선택)
  - ChatMessageResponse : id, sender_type, content, created_at, sources(List, 기본 [])
    ※ sources 필드가 없으면 대화 내역 재로드 시 출처가 사라지므로 반드시 포함해야 함


----------------------------------------------------------------
4. services/ - 비즈니스 로직
----------------------------------------------------------------

[auth.py]
  - bcrypt를 직접 사용해 비밀번호를 해싱/검증한다.
    (passlib은 bcrypt 5.x와 호환 불가로 제거됨)
    _hash(password)  → bcrypt.hashpw()
    _verify(pw, hash)→ bcrypt.checkpw()

  - check_login_id_available(login_id): 해당 login_id 존재 여부 반환 (bool)

  - register_user(): login_id·student_id 중복 검사 후 INSERT.
    응답: id(UUID), student_id, login_id, name

  - login_user(): DB에서 유저 조회 → bcrypt.checkpw()로 비밀번호 비교.
    응답에는 항상 비밀번호 해시를 제외한 UserResponse만 반환.

  - update_user(user_id, student_id, login_id, name, password):
    변경된 필드만 updates dict에 담아 UPDATE 실행.
    student_id·login_id 변경 시 다른 유저와의 중복 검사 (neq 현재 user_id).
    변경사항이 없으면 400 반환.

[category.py]
  - create_category(): categories 테이블에 INSERT.
  - get_categories(): 특정 user_id의 카테고리를 오래된 순으로 조회.
  - delete_category(): 연결된 문서가 있으면 409 에러 반환 (안전 삭제).

[document_storage.py]  ← 신규 (document.py에서 분리)
  스토리지 관련 관심사만 담당. AI/임베딩 코드 없음.

  상수:
    ALLOWED_EXTENSIONS = {".pdf"}
    CONTENT_TYPES      = {".pdf": "application/pdf"}
    MAGIC_NUMBERS      = {".pdf": b"%PDF"}

  함수:

  validate_file(original_name, contents)
    - 확장자 화이트리스트 검사 (.pdf)
    - 파일 크기 50MB 초과 검사
    - 매직 넘버(%PDF) 검사로 확장자 위조 탐지

  upload_document(original_name, contents, ext, user_id, category_id)
    - UUID로 랜덤 파일명 생성 후 Supabase Storage 'documents' 버킷에 업로드
    - documents 테이블에 메타데이터 INSERT (parsing_status = 'PENDING')
    - 스토리지 업로드 성공 후 DB 실패 시 스토리지 파일도 롤백 삭제
    - 생성된 document_id 반환

  get_documents_by_user(user_id)  : 해당 유저의 모든 문서 조회
  get_documents_by_category(id)   : 카테고리별 문서 조회
  get_document_view(document_id)  : signed_url + chunks 전체 content 반환
  delete_document(document_id)    : chunks → DB 레코드 → 스토리지 순서로 삭제

[document_rag.py]  ← 신규 (document.py에서 분리)
  RAG 파이프라인 전담. Gemini API + 임베딩 관련 코드만 담당.
  Gemini 모델/상수: core/gemini.py에서 import

  함수:

  _extract_pdf_gemini(file_bytes, filename)  ← 내부 헬퍼
    - Gemini Files API로 PDF를 업로드해 멀티모달 텍스트 추출
    - 이미지·표·차트 설명 포함, 각 페이지에 [N페이지] 마커 삽입
    - 슬라이드/페이지 제목을 "## 제목명" 형식으로 마킹 (섹션 청킹용)
    - client.files.upload(file=tmp_path, ...) 로 업로드 (path= 아님 주의)
    - 임시 파일 생성 → 업로드 → 추출 → 임시 파일·업로드 파일 모두 정리
    - 실패 시 PyMuPDF(fitz)로 폴백

  _section_aware_chunks(text, max_size)  ← 내부 헬퍼
    - "## 제목" 기준으로 섹션 분리 후 max_size(1000)까지 합산하여 청크 생성
    - 섹션이 작으면 다음 섹션과 합산 → DB 행 수 절감
    - 섹션 하나가 max_size 초과 시 고정 크기로 분할
    - ## 마커 없는 문서는 고정 크기 청킹으로 폴백

  process_document_rag(document_id)  ← 백그라운드 비동기 실행
    RAG 인제스천 파이프라인 전체:
    Step 1. Supabase Storage에서 파일 바이트 다운로드
    Step 2. _extract_pdf_gemini()로 멀티모달 텍스트 추출
            성공 → fallback_used=False / 실패 시 PyMuPDF 폴백 → fallback_used=True
    Step 3. _section_aware_chunks()로 섹션 합산 청킹 (최대 CHUNK_SIZE=1000자)
    Step 4. 각 청크를 gemini-embedding-001로 임베딩 벡터 생성
            task_type="retrieval_document", dimensions=1536
    Step 5. document_chunks 테이블에 (content, embedding, chunk_index) INSERT
    Step 6. documents 테이블 parsing_status 업데이트
            Gemini 성공 → 'COMPLETED' / PyMuPDF 폴백 → 'PARTIAL' / 오류 → 'FAILED'

[document.py]  ← 하위 호환 re-export 레이어
  실제 구현은 document_storage.py와 document_rag.py에 있음.
  이 파일을 import하던 기존 코드가 변경 없이 동작하도록 두 파일을 재노출.
  ※ 새 코드는 document_storage / document_rag에서 직접 import 권장.

[chat.py]
  Gemini 모델/상수: core/gemini.py에서 import

  보조 함수 (ask_question 분리):

  _load_history(session_id, limit=HISTORY_LIMIT)
    - chat_messages에서 최근 N개 조회 후 시간순(오래된→최신)으로 반환

  _save_message(session_id, sender_type, content, sources=None)
    - chat_messages에 INSERT, 실패 시 HTTPException(500) 발생

  _resolve_document_ids(session_id, document_ids)
    - document_ids가 None이면: 세션의 user_id 조회 → 해당 유저의 문서 ID 목록 반환
    - 세션이 없으면 HTTPException(404)
    - 프론트에서 카테고리 필터 없이 전송하면 자동으로 유저 전체 문서 검색

  _generate_embedding(text)
    - gemini-embedding-001로 텍스트 임베딩 벡터 생성
    - task_type="retrieval_query"

  _build_prompt(context, history_str, content, filenames)
    - [강의자료 내용] + [이전 대화 내역] + 질문 조합
    - AI에게 실제 참고 파일명을 [참고자료: ...] 형식으로 명시하도록 지시

  공통 헬퍼:

  _rewrite_query(content, history)
    - 팔로업 질문("저거 코드로는?")을 독립적인 검색 쿼리로 재작성
    - history가 없으면 content 그대로 반환 (불필요한 API 호출 방지)
    - 최근 REWRITE_HISTORY_WINDOW(4)개 대화만 사용
    - 실패 시 원본 content 반환 (예외 흡수)
    ※ 원래 질문의 의도 보존 규칙:
      - 지시어('그거', '이거', '거기서' 등)가 있을 때만 맥락으로 보완
      - 강의자료 커버리지 질문('자세히 안나와있지 않아?' 등)은 검색 쿼리로 바꾸지 않음
      - 질문이 이미 명확하면 그대로 반환

  get_relevant_chunks(query_vector, document_ids, limit=RAG_CHUNK_LIMIT)
    - Supabase RPC 'match_document_chunks' 호출
    - 코사인 유사도 >= 0.5인 청크를 최대 RAG_CHUNK_LIMIT(6)개 반환
    - 실패 시 에러 로깅 후 [] 반환 (graceful degradation)

  get_relevant_chunks_with_sources(query_vector, document_ids, limit)
    - get_relevant_chunks() 후 각 청크에 filename, category, page 정보 추가
    - documents·categories 테이블 조인하여 출처 메타데이터 보강

  _build_context(chunks), _extract_sources(chunks)
    - _build_context: 각 청크 앞에 "[#ref: 파일명]" 내부 식별자 라벨을 붙여 문자열 생성
      ※ "[출처: ...]" 형식은 AI가 답변에 그대로 복사하는 문제가 있어 "[#ref: ...]"로 변경
    - _extract_sources: 중복 제거된 출처 목록 반환

  _filter_used_sources(ai_answer, all_sources)
    - AI 답변 끝의 [참고자료: 파일명1|파일명2] 마커를 파싱
    - 실제로 사용된 파일만 sources로 필터링 (마커 없으면 빈 배열 반환)
    - 마커를 답변 텍스트에서 제거 후 (cleaned_answer, filtered_sources) 반환

  핵심 함수:

  (1) ask_question(session_id, content, document_ids)
      세션 기반 RAG 질의응답 파이프라인:
      Step 1. _load_history()로 과거 대화 조회 (질문 저장 전에 조회)
      Step 2. _save_message()로 사용자 질문 저장 (USER)
      Step 3. _rewrite_query()로 팔로업 질문 독립 쿼리로 변환
      Step 4. _generate_embedding()으로 벡터화
      Step 5. _resolve_document_ids()로 검색 대상 문서 결정
      Step 6. get_relevant_chunks_with_sources()로 관련 청크 검색
      Step 7. _build_prompt()로 프롬프트 조합 → Gemini로 답변 생성
      Step 8. _filter_used_sources()로 실제 참고 자료만 추출
      Step 9. _save_message()로 AI 답변 + sources 저장
      Step 10. {"message": ..., "sources": [...]} 반환
      ※ 내부 예외는 HTTPException(500)으로 변환하여 반환

  (2) ask_about_document(document_id, content)
      문서 미니 챗 RAG 질의응답 파이프라인:
      Step 1. 사용자 질문을 document_chat_messages에 저장 (USER)
      Step 2. 대화 내역 HISTORY_LIMIT개 조회
      Step 3. _rewrite_query()로 팔로업 질문 변환
      Step 4. _generate_embedding()으로 벡터화
      Step 5. get_relevant_chunks_with_sources()로 해당 문서 청크만 검색
      Step 6. 프롬프트 조합 + Gemini로 답변 생성
      Step 7. AI 답변만 document_chat_messages에 저장 (sources 미저장)
      Step 8. {"answer": ..., "sources": []} 반환
      ※ 내부 예외는 HTTPException(500)으로 변환하여 반환

  (3) delete_session(session_id)
      - chat_messages 먼저 삭제 후 chat_sessions 삭제 (FK 제약 순서 준수)


----------------------------------------------------------------
5. api/routes/ - HTTP 엔드포인트
----------------------------------------------------------------

라우터는 HTTP 요청을 받아 서비스 함수를 호출하고 응답을 반환하는 역할만 한다.
비즈니스 로직은 services/에 위임한다.

[auth.py]  prefix=/auth
  POST /auth/register              → register_user()        → 201 Created
  POST /auth/login                 → login_user()           → 200 OK
  GET  /auth/check-login-id        → check_login_id_available()
                                     ?login_id=xxx → {"available": bool}
  PUT  /auth/users/{user_id}       → update_user() (키워드 인수 사용)
                                     변경된 필드만 선택적으로 수정

[categories.py]  prefix=/categories
  POST   /categories              → create_category()
  GET    /categories/{user_id}    → get_categories()
  DELETE /categories/{id}         → delete_category()

[documents.py]  prefix=/documents
  POST   /documents/upload             → 파일 검증 → 스토리지 업로드 → RAG 백그라운드 예약
  GET    /documents/user/{user_id}     → 유저의 문서 목록 조회
  GET    /documents/category/{id}      → 카테고리별 문서 조회
  GET    /documents/{id}/view          → 문서 signed URL + 전체 텍스트 반환
  POST   /documents/{id}/ask           → ask_about_document() (문서 미니 챗)
  GET    /documents/{id}/chat          → document_chat_messages 전체 조회 (시간순)
  DELETE /documents/{id}/chat          → 해당 문서의 채팅 내역 전체 삭제 (204)
  DELETE /documents/{id}               → 문서 삭제 (chunks + DB + 스토리지)

[chat.py]  prefix=/chat
  POST   /chat/session               → chat_sessions 테이블에 INSERT
  GET    /chat/sessions/{user_id}    → 해당 유저의 세션 목록 (최신순)
  POST   /chat/ask                   → ask_question() (세션 기반 RAG 파이프라인)
  GET    /chat/messages/{session_id} → 해당 세션의 메시지 목록 (시간순, sources 포함)
  DELETE /chat/sessions/{session_id} → 메시지 먼저 삭제 후 세션 삭제


----------------------------------------------------------------
6. DB 테이블 구조
----------------------------------------------------------------

users
  id (PK, UUID, gen_random_uuid())
  student_id (UNIQUE)
  login_id (UNIQUE)
  password (bcrypt 해시)
  name

categories
  id (PK), user_id (FK→users.id), name, created_at

documents
  id (PK), user_id (FK→users.id), category_id (FK→categories.id),
  original_file_name, file_path, parsing_status, created_at
  parsing_status: 'PENDING' | 'COMPLETED' | 'PARTIAL' | 'FAILED'
  ※ PARTIAL: Gemini 추출 실패 후 PyMuPDF 폴백으로 처리된 경우 (이미지·표 인식 없음)

document_chunks
  id (PK), document_id (FK→documents), content, embedding (vector(1536)),
  chunk_index, created_at

chat_sessions
  id (PK), user_id (FK→users.id), title, created_at

chat_messages
  id (PK), session_id (FK→chat_sessions), sender_type, content,
  sources (JSONB, default []), created_at

document_chat_messages
  id (PK), document_id (FK→documents ON DELETE CASCADE),
  sender_type, content, created_at
  ※ 출처(sources)는 저장하지 않음 (보고 있는 문서가 곧 출처이므로)

  ※ UUID 마이그레이션 이력:
    기존: users.student_id를 PK로 사용, 다른 테이블이 student_id로 FK 참조
    현재: users.id(UUID)가 PK, student_id는 UNIQUE 컬럼으로만 유지
          다른 테이블의 FK는 모두 user_id(UUID)로 변경


----------------------------------------------------------------
7. 전체 데이터 흐름 요약
----------------------------------------------------------------

[문서 업로드 흐름]
  클라이언트 → POST /documents/upload
    → 파일 검증 (확장자·크기·매직넘버)
    → Supabase Storage 업로드
    → documents 테이블 INSERT (PENDING)
    → 즉시 응답 반환
    → [백그라운드] Gemini Files API로 PDF 멀티모달 추출
    → CHUNK_SIZE(1000)자 청킹 → gemini-embedding-001 임베딩
    → document_chunks INSERT → parsing_status COMPLETED

[세션 채팅 질의응답 흐름]
  클라이언트 → POST /chat/ask
    → 과거 대화 조회 → 질문 DB 저장 (USER)
    → 쿼리 재작성 (gemini-3.1-flash-lite)
    → 재작성된 쿼리 벡터화 (gemini-embedding-001)
    → 유사 청크 검색 (pgvector, 유사도 >= 0.5)
    → 프롬프트 조합 → gemini-3.1-flash-lite 답변 생성
    → 출처 필터링 → AI 답변 + sources DB 저장
    → 클라이언트에 반환

[문서 미니 챗 질의응답 흐름]
  클라이언트 → POST /documents/{id}/ask
    → 질문 document_chat_messages에 저장 (USER)
    → 최근 대화 조회 → 쿼리 재작성
    → 벡터화 → 해당 문서 청크만 검색
    → 프롬프트 조합 → gemini-3.1-flash-lite 답변 생성
    → AI 답변만 DB 저장 (sources 미저장)
    → 클라이언트에 반환

8. 주요 외부 의존성
----------------------------------------------------------------

  supabase         : DB 쿼리 + 스토리지 파일 관리
  google-genai     : 임베딩(gemini-embedding-001) + 채팅(gemini-3.1-flash-lite)
                     + 쿼리 재작성(gemini-3.1-flash-lite) + PDF 추출(Files API)
                     ※ 구버전 google-generativeai는 deprecated → google-genai로 교체
  pymupdf (fitz)   : PDF 텍스트 추출 폴백
  bcrypt           : 비밀번호 해싱 (passlib 제거 후 직접 사용)
  fastapi          : HTTP 서버 프레임워크
  uvicorn          : ASGI 서버
