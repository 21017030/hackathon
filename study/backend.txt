  스택: FastAPI + Supabase (PostgreSQL + pgvector) + Google Gemini

----------------------------------------------------------------
2. core/ - 공통 설정
----------------------------------------------------------------

[config.py]
  - python-dotenv로 .env 파일을 읽어 환경 변수를 상수로 노출한다.
  - SUPABASE_URL, SUPABASE_KEY가 없으면 앱 시작 시 RuntimeError로 즉시 종료.
  - GEMINI_API_KEY는 없어도 경고만 출력 (RAG 기능만 비활성화됨).
  - MAX_FILE_SIZE = 50MB 상수 정의.

[supabase.py]
  - create_client()를 호출해 supabase 클라이언트를 모듈 수준(전역)에서 1개만 생성.
  - 모든 service 파일에서 'from app.core.supabase import supabase'로 가져다 씀.


----------------------------------------------------------------
3. models/ - 데이터 형식 (Pydantic)
----------------------------------------------------------------

[auth.py]
  - RegisterRequest  : student_id, login_id, password, name (회원가입 입력)
  - LoginRequest     : login_id, password (로그인 입력)
  - UserResponse     : student_id, login_id, name (응답 - 비밀번호 제외)

[category.py]
  - CategoryCreate   : student_id, name (카테고리 생성 입력)
  - CategoryResponse : id, student_id, name, created_at (응답)

[document.py]
  - UploadResponse   : document_id, file_path, original_file_name (업로드 직후 응답)
  - DocumentResponse : id, student_id, category_id, original_file_name,
                       file_path, parsing_status, created_at (목록 조회 응답)

[chat.py]
  - ChatSessionCreate   : student_id, title
  - ChatSessionResponse : id, student_id, title, created_at
  - ChatAskRequest      : session_id, content, document_ids(선택)
  - ChatMessageResponse : id, sender_type, content, created_at, sources(List, 기본 [])
    ※ sources 필드가 없으면 대화 내역 재로드 시 출처가 사라지므로 반드시 포함해야 함


----------------------------------------------------------------
4. services/ - 비즈니스 로직
----------------------------------------------------------------

[auth.py]
  - passlib(bcrypt)로 비밀번호를 해싱해서 저장한다.
  - register_user(): login_id 및 student_id 중복 검사 후 INSERT.
  - login_user(): DB에서 유저 조회 → bcrypt.verify()로 비밀번호 비교.
  - 응답에는 항상 비밀번호 해시를 제외한 UserResponse만 반환.

[category.py]
  - create_category(): categories 테이블에 INSERT.
  - get_categories(): 특정 student_id의 카테고리를 오래된 순으로 조회.
  - delete_category(): 연결된 문서가 있으면 409 에러 반환 (안전 삭제).

[document.py]
  사용 모델:
    CHAT_MODEL      = "gemini-2.5-flash"       (PDF 내용 추출 + 채팅 답변)
    REWRITE_MODEL   = "gemini-3.1-flash-lite"  (쿼리 재작성 전용 경량 모델)
    EMBEDDING_MODEL = "gemini-embedding-001"   (벡터 임베딩)
    EMBEDDING_DIMENSIONS = 1536               (pgvector HNSW 인덱스 호환)

  허용 파일 형식: PDF만 허용 (.pdf)

  핵심 함수:

  (1) validate_file(original_name, contents)
      - 확장자 화이트리스트 검사 (.pdf)
      - 파일 크기 50MB 초과 검사
      - 매직 넘버(%PDF) 검사로 확장자 위조 탐지

  (2) _extract_pdf_gemini(file_bytes, filename)
      - Gemini Files API로 PDF를 업로드해 멀티모달 텍스트 추출
      - 이미지·표·차트 설명 포함, 각 페이지에 [N페이지] 마커 삽입
      - 임시 파일 생성 → 업로드 → 추출 → 임시 파일·업로드 파일 모두 정리
      - 실패 시 PyMuPDF(fitz)로 폴백

  (3) upload_document(original_name, contents, ext, student_id, category_id)
      - UUID로 랜덤 파일명 생성 후 Supabase Storage 'documents' 버킷에 업로드
      - documents 테이블에 메타데이터 INSERT (parsing_status = 'PENDING')
      - 스토리지 업로드 성공 후 DB 실패 시 스토리지 파일도 롤백 삭제
      - 생성된 document_id 반환

  (4) process_document_rag(document_id)  ← 백그라운드 비동기 실행
      RAG 인제스천 파이프라인 전체:
      Step 1. Supabase Storage에서 파일 바이트 다운로드
      Step 2. _extract_pdf_gemini()로 멀티모달 텍스트 추출
              실패 시 PyMuPDF 폴백
      Step 3. 1000자 단위로 텍스트 청킹
      Step 4. 각 청크를 gemini-embedding-001로 임베딩 벡터 생성
              task_type="retrieval_document", dimensions=1536
      Step 5. document_chunks 테이블에 (content, embedding, chunk_index) INSERT
      Step 6. documents 테이블 parsing_status를 'COMPLETED'로 업데이트
              (실패 시 'FAILED'로 업데이트)

  추가 함수:
  - get_documents_by_student(student_id): 해당 학생의 모든 문서 조회
  - get_documents_by_category(category_id): 카테고리별 문서 조회
  - get_document_view(document_id): signed_url + chunks 전체 content 반환
  - delete_document(document_id): chunks → DB 레코드 → 스토리지 순서로 삭제

[chat.py]
  공통 헬퍼 함수:

  _rewrite_query(content, history)
    - 팔로업 질문("저거 코드로는?")을 독립적인 검색 쿼리로 재작성
    - history가 없으면 content 그대로 반환 (불필요한 API 호출 방지)
    - gemini-3.1-flash-lite(경량 모델) 사용으로 응답 속도 및 비용 절감
    - 실패 시 원본 content 반환 (예외 흡수)

  get_relevant_chunks(query_vector, document_ids, limit=6)
    - Supabase RPC 'match_document_chunks' 호출
    - 코사인 유사도 >= 0.5인 청크를 최대 6개 반환
    - document_ids를 넘기면 해당 문서들로 검색 범위 필터링

  get_relevant_chunks_with_sources(query_vector, document_ids, limit)
    - get_relevant_chunks() 호출 후 각 청크에 filename, category, page 정보 추가
    - documents 테이블과 categories 테이블을 조인하여 출처 메타데이터 보강

  _build_context(chunks)
    - [출처: 카테고리 > 파일명 N페이지] 형식으로 청크 묶어 프롬프트용 문자열 생성

  _extract_sources(chunks)
    - 파일명 기준으로 중복 제거한 출처 목록 반환
    - 반환 형식: [{"filename": ..., "category": ...}]

  _filter_used_sources(ai_answer, all_sources)
    - AI 답변 끝의 [참고자료: 파일명1|파일명2] 마커를 파싱
    - 실제로 사용된 파일만 sources로 필터링
    - 마커를 답변 텍스트에서 제거 후 (cleaned_answer, filtered_sources) 반환

  핵심 함수:

  (1) ask_question(session_id, content, document_ids)
      세션 기반 RAG 질의응답 파이프라인:
      Step 1. 사용자 질문을 chat_messages에 'USER' 타입으로 INSERT
      Step 2. 최근 대화 6개 DB에서 조회 (쿼리 재작성에 활용)
      Step 3. _rewrite_query()로 팔로업 질문을 독립 쿼리로 변환
      Step 4. 변환된 쿼리를 gemini-embedding-001로 벡터화
              task_type="retrieval_query"
      Step 5. get_relevant_chunks_with_sources()로 관련 청크 최대 6개 검색
      Step 6. 프롬프트 구성: [강의자료 내용] + [이전 대화] + [질문]
              AI에게 사용한 파일명을 [참고자료: ...] 형식으로 명시하도록 지시
      Step 7. gemini-2.5-flash로 답변 생성
      Step 8. _filter_used_sources()로 실제 참고 자료만 추출
      Step 9. sources를 JSONB로 포함해 chat_messages에 'AI' 타입으로 INSERT
      Step 10. {"message": ..., "sources": [...]} 반환

  (2) ask_about_document(document_id, content)
      문서 미니 챗 RAG 질의응답 파이프라인:
      Step 1. 사용자 질문을 document_chat_messages에 'USER' 타입으로 INSERT
      Step 2. 최근 대화 6개 document_chat_messages DB에서 조회
      Step 3. _rewrite_query()로 팔로업 질문 변환
      Step 4. gemini-embedding-001로 벡터화
      Step 5. get_relevant_chunks_with_sources()로 해당 문서 청크만 검색
      Step 6. 프롬프트 구성 + gemini-2.5-flash로 답변 생성
      Step 7. _filter_used_sources()로 출처 필터링
      Step 8. AI 답변을 sources와 함께 document_chat_messages에 INSERT
      Step 9. {"answer": ..., "sources": [...]} 반환

  (3) delete_session(session_id)
      - chat_messages 먼저 삭제 후 chat_sessions 삭제 (FK 제약 순서 준수)

5. api/routes/ - HTTP 엔드포인트
----------------------------------------------------------------

라우터는 HTTP 요청을 받아 서비스 함수를 호출하고 응답을 반환하는 역할만 한다.
비즈니스 로직은 services/에 위임한다.

[auth.py]  prefix 없음
  POST /auth/register  → services/auth.register_user()   → 201 Created
  POST /auth/login     → services/auth.login_user()      → 200 OK

[categories.py]  prefix=/categories
  POST   /categories              → services/category.create_category()
  GET    /categories/{student_id} → services/category.get_categories()
  DELETE /categories/{id}         → services/category.delete_category()

[documents.py]  prefix=/documents
  POST   /documents/upload             → 파일 검증 → 스토리지 업로드 → RAG 백그라운드 예약
  GET    /documents/student/{id}       → 학생의 문서 목록 조회
  GET    /documents/category/{id}      → 카테고리별 문서 조회
  GET    /documents/{id}/view          → 문서 signed URL + 전체 텍스트 반환
  POST   /documents/{id}/ask           → ask_about_document() (문서 미니 챗)
  GET    /documents/{id}/chat          → document_chat_messages 전체 조회 (시간순)
  DELETE /documents/{id}/chat          → 해당 문서의 채팅 내역 전체 삭제 (204)
  DELETE /documents/{id}               → 문서 삭제 (chunks + DB + 스토리지)

[chat.py]  prefix=/chat
  POST   /chat/session               → chat_sessions 테이블에 INSERT
  GET    /chat/sessions/{student_id} → 해당 학생의 세션 목록 (최신순)
  POST   /chat/ask                   → ask_question() (세션 기반 RAG 파이프라인)
  GET    /chat/messages/{session_id} → 해당 세션의 메시지 목록 (시간순, sources 포함)
  DELETE /chat/sessions/{session_id} → 메시지 먼저 삭제 후 세션 삭제


----------------------------------------------------------------
6. DB 테이블 구조
----------------------------------------------------------------

users
  student_id (PK), login_id (UNIQUE), password, name

categories
  id (PK), student_id (FK→users), name, created_at

documents
  id (PK), student_id (FK→users), category_id (FK→categories),
  original_file_name, file_path, parsing_status, created_at
  parsing_status: 'PENDING' | 'COMPLETED' | 'FAILED'

document_chunks
  id (PK), document_id (FK→documents), content, embedding (vector),
  chunk_index, created_at

chat_sessions
  id (PK), student_id (FK→users), title, created_at

chat_messages
  id (PK), session_id (FK→chat_sessions), sender_type, content,
  sources (JSONB, default []), created_at

document_chat_messages
  id (PK), document_id (FK→documents ON DELETE CASCADE),
  sender_type, content, sources (JSONB, default []), created_at


----------------------------------------------------------------
7. 전체 데이터 흐름 요약
----------------------------------------------------------------

[문서 업로드 흐름]
  클라이언트 → POST /documents/upload
    → 파일 검증 (확장자·크기·매직넘버)
    → Supabase Storage 업로드
    → documents 테이블 INSERT (PENDING)
    → 즉시 응답 반환
    → [백그라운드] Gemini Files API로 PDF 멀티모달 추출
    → 1000자 청킹 → gemini-embedding-001 임베딩
    → document_chunks INSERT → parsing_status COMPLETED

[세션 채팅 질의응답 흐름]
  클라이언트 → POST /chat/ask
    → 질문 DB 저장 (USER)
    → 최근 대화 6개 조회 → 쿼리 재작성 (gemini-3.1-flash-lite)
    → 재작성된 쿼리 벡터화 (gemini-embedding-001)
    → 유사 청크 검색 (pgvector, 유사도 >= 0.5)
    → 프롬프트 조합 → gemini-2.5-flash 답변 생성
    → 출처 필터링 → AI 답변 + sources DB 저장
    → 클라이언트에 반환

[문서 미니 챗 질의응답 흐름]
  클라이언트 → POST /documents/{id}/ask
    → 질문 document_chat_messages에 저장 (USER)
    → 최근 대화 6개 DB에서 로드 → 쿼리 재작성
    → 벡터화 → 해당 문서 청크만 검색
    → 프롬프트 조합 → gemini-2.5-flash 답변 생성
    → 출처 필터링 → AI 답변 + sources DB 저장
    → 클라이언트에 반환


----------------------------------------------------------------
8. 주요 외부 의존성
----------------------------------------------------------------

  supabase         : DB 쿼리 + 스토리지 파일 관리
  google-genai     : 임베딩(gemini-embedding-001) + 채팅(gemini-2.5-flash)
                     + 쿼리 재작성(gemini-3.1-flash-lite) + PDF 추출(Files API)
                     ※ 구버전 google-generativeai는 deprecated → google-genai로 교체
  pymupdf (fitz)   : PDF 텍스트 추출 폴백
  passlib[bcrypt]  : 비밀번호 해싱
  fastapi          : HTTP 서버 프레임워크
  uvicorn          : ASGI 서버
